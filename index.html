《图解Http》读书笔记

《图解Http》是本很通俗的书，萌系插画让人很容易理解，日本作家的风格非常明显。本文记录下我认为比较重要的知识点，留日后复习。

1.网络基础

HTTP1.0 vs HTTP1.1

目前主流的是HTTP1.1

TCP/IP

通常使用的网络，包括互联网都是在TCP/IP协议族基础上运作的，把与互联网相关的协议集合起来总称为TCP/IP。TCP/IP分为四层：应用层，传输层，网络层，数据链路层。

  • 应用层 FTP DNS HTTP
  • 传输层 TCP UDP
  • 网络层
  • 数据链路层
URI URL

Uniform Resource Identifier

Uniform Resource Locator

2.简单的HTTP

HTTP/1.1中可以使用的方法
  • GET 获取资源
  • POST 传输实体主体
  • PUT 传输文件
  • HEAD 获得报文首部
  • DELETE 删除文件
  • OPTIONS (only in 1.1) 询问支持的方法
  • TRACE (only in 1.1) 追踪路径
  • CONNECT (only in 1.1) 要求用隧道协议连接代理
HTTP 的持久连接

只要任意一端没有明确提出断开连接，则保持TCP连接状态。
建立TCP连接：三次握手 断开TCP连接：四次挥手。

管线化（pipelining）：管线化技术能够做到并行发送多个请求，不需要一个一个等待相应。

Cookie

HTTP是无状态协议，为了记录状态引入了Cookie信息来管理客户端的状态。

HTTP 返回状态码

状态码 含义
200 ok 成功
204 no content 请求处理成功，没有新资源需要返回
204 partcial content 成功执行了对某部分资源的请求

304 not modified 资源不符合请求条件
307 temporaryly redirect 临时重定向
400 bad requst 请求有错
401 unauthorized 需要认证
403 forbidden 拒绝访问
404 not found 无法找到请求的资源

503 service unavailable 服务器无法响应

【总结篇】所有常见的排序算法

虽然排序是入门的算法，但是也是很有必要总结一下的，距离第一次学习排序算法已经快四年了。而Java提供了Aarrys.sort()方法让我基本上没有再手写过排序算法。大一学习数据结构，接触到了冒泡排序，当时让我手写冒泡还很费劲。快排愣是看了半天都没懂……

出现在这片总结里的排序算法有一下几种：

  • 冒泡排序
  • 插入排序
  • 归并排序
  • 堆排序
  • 快速排序
  • （其他待续）

冒泡排序

冒泡排序的基本思想是，通过一次遍历，找出一个当前最大（最小）值，排在最后（最前）。是一种原地排序的方法，其时间复杂度是O(n^2)。

public void bubble_sort(int[] A) {
    for (int i = 0; i < A.length; i++) {
        for (int j = 1; j < A.length - i; j++) {
            if (A[j] < A[j - 1]) {
                //swap
                int temp = A[j];
                A[j] = A[j-1];
                A[j-1] = temp;
            }
        }
    }
}

插入排序

插入排序是把排序的过程想象为打扑克时整理手牌的过程，假设手中的牌已经是排序好的，那么将一张新的手牌插入已排序好的手牌，这过程就是插入排序。插入排序的时间复杂度也是O(n^2)。

public void insert_sort(int[] A) {
    for (int i = 1; i < A.length; i++) {
        int j = i;
        while (j > 0 && A[j] < A[j - 1]) {
            int temp = A[j - 1];
            A[j - 1] = A[j];
            A[j] = temp;
            j--;
        }
    }
}

归并排序

这是一个时间复杂度为O(nlgn)的排序算法。其本质思想是分治法：Divide and conquer。

将一个大问题分解为最小的问题，再将其合并为大问题的解。在归并排序中，最小的问题即两个数的排序 a>b or a<b or a==b。解决了最小的问题后，下一步是将最小的问题合并，即：合并两个已排序的数组。最终得到一个排序好的完整数组。通过递归可以顺利的解决该问题~

public void merge_sort(int[] A, int p, int r) {
    if (p < r) {
        int q = (p + r) / 2;
        merge_sort(A, p, q);
        merge_sort(A, q + 1, r);
        merge(A, p, q, r);
    }
}

public void merge(int[] A, int p, int q, int r) {
    int n1 = q - p + 1;
    int n2 = r - q;
    int[] L = Arrays.copyOfRange(A, p, q + 1);
    int[] R = Arrays.copyOfRange(A, q + 1, r + 1);
    int i = 0, j = 0, k = p;
    while (i < n1 && j < n2) {
        if (L[i] < R[j]) {
            A[k++] = L[i];
            i++;
        } else {
            A[k++] = R[j];
            j++;
        }
    }
    while (i < n1) {
        A[k++] = L[i];
        i++;
    }
    while (j < n2) {
        A[k++] = R[j];
        j++;
    }
}

第一次调用的话就是：merge_sort(A,0,A.length-1)

堆排序

堆可以被视为一棵完全二叉树，用数组存储，故得知某个节点的下标后，很容易得知其父母，左儿子和有儿子的节点下标。

// parent: i/2
// left child: i*2
// right child: i*2+1

对于堆来说，当堆建成之后，每次获取最大值（对于大顶堆而言）的时间复杂度都是O(1)，因为（大顶堆）可以保证永远都在第一个下标。

对于堆排序来说，其步骤大概是这样的：首先将无序数组构造成一个大顶堆，每次拿去顶点的最大值，放在队尾，继续将前n-1个数构造大顶堆。

public void heap_sort(int[] A) {
    build_max_heap(A);
    int heap_size = A.length;
    for (int i = A.length - 1; i > 0; i--) {
        int temp = A[i];
        A[i] = A[0];
        A[0] = temp;
        heap_size--;
        max_heapify(A, 0, heap_size);
    }
}

public void build_max_heap(int[] A) {
    for (int i = A.length / 2; i >= 0; i--) {
        max_heapify(A, i, A.length);
    }
}

public void max_heapify(int[] A, int i, int heap_size) {
    if (i >= 0) {
        int left = ((i + 1) << 1) - 1;
        int right = left + 1;
        int largest = i;
        if (left < heap_size && A[i] < A[left]) {
            largest = left;
        }
        if (right < heap_size && A[largest] < A[right]) {
            largest = right;
        }
        //swap
        if (i != largest) {
            int temp = A[i];
            A[i] = A[largest];
            A[largest] = temp;
            max_heapify(A, largest, heap_size);
        }
    }
}

max_heapify是对最大堆进行操作的子程序，其目的是保持最大堆的性质。堆排序的时间代价为O(nlgn),其中调用build_max_heap的时间为O(n),n-1次max_heapify调用中每次的时间代价为O(lgn)

快速排序

快速排序是用于排序的最佳实用选择，因为其平均性能相当好（虽然其最坏的运行时间为O(n^2)）。
快速排序也是基于分治法的思想，对于一个无序的数组，将最后一个元素作为主元，通过partition将其就地重排，排序结果是主元位于数组中间，主元左边的元素都小于等于主元，右边的元素都大于等于主元。然后对主元左边和右边的子数组分别再次调用partition。直到子数组无可划分，排序停止。

public void quick_sort(int[] A, int p, int r) {
    if (p < r) {
        int i = p - 1;
        int x = A[r];
        for (int j = p; j < r; j++) {
            if (A[j] <= x) {
                i++;
                int temp = A[i];
                A[i] = A[j];
                A[j] = temp;
            }
        }
        int temp = A[i + 1];
        A[i + 1] = x;
        A[r] = temp;
        quick_sort(A, p, i);
        quick_sort(A, i + 1, r);
    }
}

背包问题总结

背包问题是动态规划的基础问题之一。最基础的三个背包问题是：

  • 01背包
  • 部分背包
  • 完全背包

01背包

01背包描述了这样一个问题：一组物品，其重量分别是 w[i], 价值分别是 v[i], 现有一个承重为 w_total 的背包，求在不超过背包容量的情况下，所容纳的物品价值的最大值是多少，每种物品有且只有一个，不能分割。

首先可能会被错误的贪心思路所带跑偏，比如“每次放入重量最小的物品”，“每次放入价值最大的物品”，“每次放入单位价值最大的物品”。事实上这三种思路都无法获得最优解，其反例显而易见。

所以需要用动态规划来解决这个问题。为什么动态规划可以解决该问题呢，因为该问题具有一个特征：最优子结构。

背包价值为j的问题，可以通过背包价值 <j 的子问题求解

如此便可以得到初步的状态转移方程：

f[i,j] = Max{f[i-1,j],f[i-1,j-wi]+vi}

其中 f[i,j] 表示在前 i 件物品中选取若干件放入大小为j的背包中，所能取得的最大价值。

但是此方法需要的空间复杂度略高，通过模拟一次背包问题的求结果过程可知，f[i,j]仅依赖于f[i-1,j]。

可以优化空间复杂度，得到新的状态转移方程

f[v] = Max{f[v],f[v-wi]+pi}

最终代码如下：

 public void package01(int[] v, int[] w, int package_w) {
    int[] dp = new int[package_w + 1];
    for (int i = 0; i < v.length; i++) {
        for (int j = package_w; j >= w[i]; j--) {
            dp[j] = Math.max(dp[j], dp[j - w[i]] + v[i]);
        }
    }
    System.out.println(dp[package_w]);
}

部分背包

虽然贪心算法不能解决01背包的问题，但是可以解决部分背包的问题。
部分背包的问题描述和01类似，但区别是可以分割。

如此便可以用贪心算法解决部分背包问题，即每次选取单位价值最大的物品。

代码不再赘述。但是需要注意的是，每个贪心算法都可以用一个更为复杂的动态规划算法来描述。但动态规划问题不一定能用贪心法来解决。

完全背包

完全背包描述的问题也与01问题类似，但是其不同在于，每种物品可以取若干个。

根据01问题类似的思路可以得到状态转移方程：

f[i,j] = Max{f[i,j-wi]+vi,f[i-1,j]}

这个状态转移方程可能比较难以理解，但是可以发现与01背包问题的状态方程只差了一点，其实就可以理解为无限制的01背包问题，f[i,j-wi]+vi的含义就表示了在可能已经包含了物品i的基础上再选一个。

优化状态方程可得到

f[v] = Max{f[v],f[v-wi]+pi}

你会发现这个状态转移方程不是跟01背包一样吗？
先看看代码

public void package01(int[] v, int[] w, int package_w) {
    int[] dp = new int[package_w + 1];
    for (int i = 0; i < v.length; i++) {
        for (int j = w[i]; j <= package_w; j++) {
            dp[j] = Math.max(dp[j], dp[j - w[i]] + v[i]);
        }
    }
    System.out.println(dp[package_w]);
}

可以发现与01背包的代码只有第二个循环的次序不同，由倒序变为了正序，可以这么理解：01背包的倒序是为了确保对于j[i-1,j]的依赖，也就是每次求f[v-wi]的时候都是不包含物品i的。而正序之后计算f[v-wi]时则都是在可能选取了物品i的基础上。

修改xib无效，运行后修改没有显示的问题

曾经被问到这样一个问题，明明修改了xib文件，但是运行之后却依然显示修改之前的xib，甚至删除了xib中的某个控件，也依然会显示出来。

试了Clean Project，删除模拟器/真机中的app，依然无用。此时我想到可能是xcode的缓存问题，打开Xcode->Preference,找到缓存地址

打开缓存文件夹，找到对应项目的文件夹删除之，重新运行，ok了。

有时候很多问题都是缓存文件搞得鬼。但是这种xib不能修改一般不会出现，遇到了两次都是在xcode版本不同的时候进行协同开发时出现的。

而且xcode的缓存文件很大，如果工程很多可以定期删除一次。
